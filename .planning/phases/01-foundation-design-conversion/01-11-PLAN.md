<objective>
Faithful page-by-page rewrite of all 4 pages + shared components to match the design-3 HTML files exactly.

This plan exists because the initial Phase 1 execution produced pages that diverged significantly from the design HTML. Multiple rounds of patching made things worse due to worktree merge issues that repeatedly reverted files. This plan does a clean, verified rewrite of each page.

**Root cause of divergence:** Executor agents "vibe-coded" the pages instead of faithfully converting the design HTML. Worktree cherry-picks had stale base commits that deleted files on merge. CSS variable syntax (`bg-[--var]`) doesn't work in Tailwind v4 — must use `bg-var` with `@theme` mapping.

**Approach:** Sequential, one page at a time, verified in the browser before committing. NO worktrees. NO parallel agents. Read the design HTML, convert faithfully, check in browser, commit.
</objective>

<lessons_learned>
1. NEVER use bg-[--custom-var] in Tailwind v4 — must use bg-custom-var with @theme mapping
2. NEVER use worktrees on this Windows setup — cherry-picks create stale base commits that delete files
3. ALWAYS verify in browser before committing — blind code changes cause regressions
4. ALWAYS read the FULL design HTML file before converting — don't assume or improvise
5. Use npx next dev (not next start) for hot-reloading during iteration
6. Do one page at a time, verify, commit — not parallel execution
</lessons_learned>
